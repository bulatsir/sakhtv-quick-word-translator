# SakhTV Quick Word Translator — дизайн

**Дата:** 2026-07-06
**Статус:** утверждён; ревью выполнено агентом (пользователь не читает JS),
все замечания ревью внесены (v2 спеки).

## Цель

Chrome-расширение для sakh.tv: при наведении курсора на слово в субтитрах
показывать его перевод с английского на русский во всплывающей подсказке.
Аналог easysubs.cc, но только для sakh.tv и только с минимальной функцией
«навёл — увидел перевод».

## Контекст (результаты разведки sakh.tv)

- Плеер: **JW Player 8.39** (`window.jwplayer` доступен на странице).
- Субтитры рендерятся текстом в DOM: `.jw-captions .jw-text-track-cue`
  (потомковый селектор: между контейнером и cue возможны промежуточные узлы).
- Страницы просмотра: `https://sakh.tv/watch/<show>/<season>/<episode>/<voice>/`.
- Наличие субтитров зависит от выбранной озвучки (у оригинальных дорожек
  они есть, у большинства дубляжей — нет).
- Требуется авторизация на сайте, но расширение работает поверх уже
  открытой пользователем страницы — самому расширению авторизация не нужна.
- **Важно:** штатный скин JW Player 8 задаёт `pointer-events: none` на
  `.jw-captions` — мышь «проходит сквозь» субтитры. Без переопределения
  (см. styles.css) hover не сработает вовсе.

## Решения, принятые с пользователем

| Вопрос | Решение |
|---|---|
| Направление перевода | en → ru, фиксировано |
| Источник перевода | Бесплатный endpoint Google Translate (`translate.googleapis.com/translate_a/single?client=gtx`) |
| Скорость | Предперевод всех слов реплики при её появлении + постоянный кэш |
| Объём v1 | Только перевод слова по наведению. Без паузы, без перевода фраз, без сохранения слов |
| Распространение | Личное пользование, загрузка через «Load unpacked» |

## Архитектура

4 файла, Manifest V3, без сборщиков и зависимостей:

```
manifest.json   — конфигурация расширения
content.js      — работа с субтитрами на странице
background.js   — service worker: переводы и кэш
styles.css      — тултип и подсветка слова
```

### Сквозное требование безопасности

Текст субтитров и ответы Google — внешние данные. Любая вставка текста в
DOM выполняется только через `document.createElement` + `textContent`.
`innerHTML` / `insertAdjacentHTML` с внешними строками запрещены.

### manifest.json

- `manifest_version: 3`.
- `content_scripts`: `content.js` + `styles.css`,
  `matches: ["https://sakh.tv/watch/*", "https://www.sakh.tv/watch/*"]`,
  `run_at: document_idle`.
- `background.service_worker: background.js`.
- `permissions: ["storage"]`.
- `host_permissions: ["https://translate.googleapis.com/*"]` — fetch к Google
  выполняется из service worker'а, чтобы не упереться в CORS со страницы.

### content.js

Обязанности:

1. **Поиск контейнера субтитров.** Плеер инициализируется асинхронно:
   MutationObserver на `document.body` ждёт появления `.jw-captions`
   без таймаута (вкладка могла быть открыта в фоне, плеер может стартовать
   поздно). Пока контейнера нет — расширение бездействует. Диагностика:
   если через 60 с контейнер не найден, один раз логируется
   `[QWT] captions container not found (yet)` — поиск при этом продолжается.
2. **Наблюдение.** MutationObserver (childList + subtree + characterData)
   вешается на **стабильного предка** — контейнер `.jwplayer` (или
   `document.body`, если так проще), а не на сам `.jw-captions`: плеер может
   пересоздать узел субтитров при смене озвучки/эпизода без перезагрузки
   страницы, и observer на нём молча умер бы. Cue-элементы находятся в
   обработчике селектором `.jw-captions .jw-text-track-cue`.
3. **Оборачивание слов.** Для каждого текстового узла cue: разбить по
   пробелам, каждое слово обернуть в `<span class="qwt-word">`
   (createElement + textContent). Нормализация слова для перевода/кэша:
   нижний регистр; срезаются только ведущие и замыкающие знаки пунктуации;
   внутренние `'` / `’` / `-` сохраняются (`don't`, `re-read`); типографский
   `’` нормализуется в `'`. Видимый текст остаётся как в оригинале, включая
   пунктуацию. Защита от зацикливания: обработанный cue помечается
   `dataset.qwtDone` **до** любых мутаций, обработчик помеченные пропускает.
4. **Предперевод.** Сразу после оборачивания — одно сообщение
   `{type:"translate", words:[...]}` в background.js со всеми
   нормализованными словами реплики. Ответ (`{слово: перевод|null}`)
   складывается в локальную карту content.js. Ответ не блокирует UI.
5. **Hover и тултип.** Делегированные `mouseover`/`mouseout` на контейнере
   плеера (`mouseenter` не всплывает и для делегирования не годится):
   - over на `.qwt-word` → взять перевод из локальной карты; если слова нет
     или значение `null` — показать «…», отправить одиночное сообщение
     `{type:"translate", words:[слово]}` и по ответу обновить тултип,
     **только если это слово всё ещё под курсором**;
   - out → скрыть тултип.

   Тултип — один переиспользуемый `<div class="qwt-tooltip">`, добавленный
   **внутрь контейнера плеера** (`.jwplayer`), а не в `body` — иначе он
   невидим в полноэкранном режиме. Позиционирование: `position: absolute`
   относительно `.jwplayer`; координаты слова берутся из
   `getBoundingClientRect()` слова **минус** rect контейнера плеера.
   По умолчанию тултип над словом; если слово у верхнего края контейнера —
   под словом.
6. **Изоляция ошибок.** Весь код content.js обёрнут так, чтобы любое
   исключение логировалось в консоль с префиксом `[QWT]` и не ломало плеер.

### background.js

Обязанности:

1. Принимает сообщения `{type: "translate", words: [...]}` из content.js.
   **Обработчик `onMessage` возвращает `true` и отвечает через
   `sendResponse` асинхронно** (иначе канал закроется до завершения fetch и
   content.js получит `undefined`).
2. Кэш двухуровневый:
   - **в памяти** (`Map`) — на время жизни service worker'а (после ~30 с
     простоя воркер засыпает и Map теряется — это нормально, storage
     компенсирует);
   - **`chrome.storage.local`** — постоянный. Ключ — слово, значение —
     перевод. **Кэшируются только непустые строки**: `null`/пустые ответы не
     записываются ни в один уровень (иначе слово навсегда осталось бы без
     перевода). Лимит storage.local (10 МБ) при словарном запасе сериалов
     недостижим, очистка не нужна.
3. Перевод — **строго по одному слову на запрос**:
   `https://translate.googleapis.com/translate_a/single?client=gtx&sl=en&tl=ru&dt=t&q=<word>`.
   Batched-вариант «слова через перевод строки» запрещён: endpoint не
   гарантирует сегментацию ответа, рассинхрон тихо испортил бы постоянный
   кэш. Ответ endpoint'а — вложенный JSON-массив; перевод берётся из
   `data[0][0][0]`, при неожиданной структуре — `null`.
4. Защита от штормов и rate limit:
   - карта «слово → in-flight Promise»: одно слово никогда не запрашивается
     параллельно дважды (наблюдатель может прислать одну реплику дважды);
   - параллелизм fetch ограничен (3–4 одновременных, остальные в очередь);
   - HTTP 429 и любые ошибки сети/HTTP → ответ `{word: null}` без ретраев;
     повторный hover пользователя запустит новый запрос.
5. Отвечает на каждое сообщение картой `{word: translation|null}`.

### styles.css

- **`.jw-captions, .jw-captions * { pointer-events: auto; }`** —
  переопределение штатного `pointer-events: none` JW Player; без этого
  hover не работает (см. Контекст). Осознанный побочный эффект: клик по
  области субтитров больше не ставит видео на паузу — для v1 приемлемо.
- `.qwt-word:hover` — лёгкая подсветка (подчёркивание/фон), чтобы было
  видно, на что наведён курсор.
- `.qwt-tooltip` — тёмная плашка с белым текстом, `position: absolute`,
  `z-index` выше контролов плеера, `pointer-events: none` (чтобы тултип не
  перехватывал мышь и не вызывал мерцание).

## Поток данных

```
JW Player вставляет реплику
  → MutationObserver (content.js)
  → оборачивание слов в span
  → message "translate" со всеми словами → background.js
  → кэш / Google Translate → ответ {слово: перевод|null} → локальная карта
  → пользователь наводит на слово
  → тултип из локальной карты; при промахе — «…» + одиночный запрос
```

## Тестирование

Ручное, на sakh.tv (автотестов в v1 нет — проект персональный, вся
сложность в интеграции с живым плеером). Порядок важен — первым проверяется
блокер-риск:

1. **Hover реально срабатывает** (pointer-events переопределён; контролы и
   оверлеи плеера не перехватывают мышь над зоной субтитров — если
   перехватывают, разбираться по факту с z-index/pointer-events).
2. Серия с оригинальной озвучкой и субтитрами: слова подсвечиваются, тултип
   показывает перевод, задержки не ощущаются.
3. Реплики сменяются — обёртка работает на каждой новой реплике.
4. Смена озвучки/эпизода **без перезагрузки страницы** — расширение
   продолжает работать (observer на стабильном предке).
5. Полноэкранный режим — тултип виден.
6. Перемотка, пауза, смена качества — расширение не ломает плеер.
7. Офлайн/ошибка сети — тултип показывает «…», плеер работает.
8. Страница без субтитров — расширение бездействует, ошибок в консоли нет.
9. Слова с пунктуацией: `don't`, `re-read`, `"Ran!"`, типографский `’`.

## Явно за рамками v1

- Пауза видео при наведении (кандидат №1 на v2). Замечание: content script
  живёт в изолированном мире и не видит `window.jwplayer` страницы —
  понадобится инъекция в MAIN world (`world: "MAIN"` или
  `chrome.scripting.executeScript`), т.е. это не «пара строк».
- Перевод всей фразы целиком.
- Сохранение слов в список «на выучить».
- Другие языковые пары и автоопределение языка.
- Публикация в Chrome Web Store.

## Риски

| Риск | Оценка |
|---|---|
| Неофициальный endpoint Google перестанет работать / 429 при всплесках | Смягчено: постоянный кэш, лимит параллелизма, отсутствие ретраев. Замена endpoint'а — одно место в background.js |
| sakh.tv сменит плеер или классы | Починка — правка селекторов в content.js |
| Субтитры конкретной дорожки вшиты в видео | Расширение бессильно; пользователь выбирает дорожку с текстовыми субтитрами |
| Контролы плеера перехватывают мышь над субтитрами | Проверяется первым пунктом ручного теста; решение по факту (z-index/pointer-events) |
